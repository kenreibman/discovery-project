## Page Contracts

### Case Detail Page (`/case/[id]`) -- Extended

**Layout:** Unchanged. Same protected layout with sidebar. Main content max-width 640px centered. 48px top padding.

**Existing content (unchanged):** Back link, case name heading, case metadata, documents section with document list, danger zone.

**New content inserted between documents section and danger zone:**

For each document with `type === "discovery_request"` that has been processed, an Extracted Requests section appears below the document list.

#### Extracted Requests Section

**Position:** Below the document list, above the danger zone. 24px top margin from last document row.

**Section header row (horizontal flex, space-between):**
- Left side:
  - "Extracted Requests" heading (14px, weight 600, text-primary)
  - 8px horizontal gap
  - Discovery sub-type badge: "RFP" or "Interrogatory" (badge component, secondary variant, 12px text, uppercase). Clickable -- opens select dropdown for override (D-14). Only visible after extraction completes.
  - 8px horizontal gap
  - Request count in parentheses: "(33)" (14px, weight 400, text-secondary)
- Right side:
  - "Re-extract" button (ghost variant, 14px): lucide `RefreshCw` icon (14px) + "Re-extract" text. Text-secondary color. Disabled with spin animation on icon during re-extraction.

**Request list (12px top margin from header):**

Vertical stack of request rows with 4px gap between rows.

**Each request row:**
- Container: `rounded` border (1px `--border`), padding 12px. Cursor pointer. Background white (`--card`).
- Hover state: background transitions to `--secondary` (#F0EDE6).
- Contents (horizontal flex, items-start, 8px gap):
  1. Expand indicator (16px width, only if text exceeds 100 characters):
     - Collapsed: lucide `ChevronRight` (16px, text-secondary), positioned at top with 2px top offset for alignment with text baseline
     - Expanded: lucide `ChevronDown` (16px, text-secondary)
     - If text is 100 characters or fewer: 16px empty spacer (maintains alignment)
  2. Request number: "{N}." (14px, weight 500, text-secondary). Right-aligned in a 24px-wide column.
  3. Request text:
     - Collapsed (default): first 100 characters + "..." if longer (14px, weight 400, text-primary). Single line with overflow hidden.
     - Expanded: full text (14px, weight 400, text-primary). Natural line wrapping. No max-height.

**Click behavior:** Clicking anywhere on the request row toggles between collapsed and expanded. Only rows with text exceeding 100 characters are expandable. Short-text rows show full text always with no chevron.

**Generate Responses button:**
- Position: 16px below the last request row. Full width of the request list container.
- Visual: Primary button variant, accent background (#C8653A), white text "Generate Responses" (16px, weight 600), 44px height.
- State: **Disabled** in Phase 3. Opacity 50%. Cursor not-allowed. Tooltip: "Coming soon".
- Becomes functional in Phase 4.

#### Discovery Sub-Type Override

**Trigger:** Click the sub-type badge in the Extracted Requests section header.

**Dropdown contents:**
- "RFP" option
- "Interrogatory" option

**Behavior:** Selection updates the document sub-type via server action. Badge text updates immediately (optimistic). Same pattern as the existing document type override select in the document list (D-14).

### Extraction Loading State

**When:** After a discovery_request document finishes uploading and classifying, extraction begins automatically (D-02). Also triggered when "Re-extract" is clicked.

**Upload flow integration:** After `classifyDocument()` succeeds and document type is `discovery_request`, the file row in the pending uploads list shows status "extracting" before being removed. The case detail page then shows the extraction loading state in the Extracted Requests section.

**Visual (replaces the request list area):**

Section header appears immediately with:
- "Extracted Requests" heading (14px, weight 600)
- No badge, no count (not yet known)
- "Re-extract" button disabled

Below the header, 12px margin:
- 3 skeleton placeholder rows, each:
  - Container: rounded border (1px `--border`), padding 12px, same dimensions as real request rows
  - Inside: skeleton shimmer bar, 80% width, 14px height, rounded
  - 8px gap below first bar, second skeleton bar at 60% width, 14px height, rounded
  - Pulse animation (`animate-pulse` from shadcn skeleton)
- Below skeleton rows: small status line (14px, text-secondary, centered): lucide `Loader2` (14px, spinning, accent color) + "Extracting requests..." text

**Duration expectation:** 10-45 seconds depending on document length. No timeout indicator shown to user.

---

### Extraction Error State

**When:** Claude API call fails, or returns malformed data despite structured output, or network error during PDF fetch from Blob.

**Visual (replaces the request list area):**

Section header appears with:
- "Extracted Requests" heading (14px, weight 600)
- No badge, no count

Below the header, 12px margin:
- Error container: rounded border (1px `--border`), padding 24px, centered content
  - lucide `AlertCircle` icon (24px, destructive #DC2626), centered
  - 8px below icon: error message (14px, weight 400, text-primary, centered, max-width 400px): "Could not extract requests from this PDF. The document may be a poor-quality scan." (D-07, verbatim)
  - 16px below message: two buttons side by side, centered, 8px gap:
    - "Retry" button (outline variant, 14px): lucide `RefreshCw` icon (14px) + "Retry" text
    - "Upload new" button (outline variant, 14px): lucide `Upload` icon (14px) + "Upload new" text

**Retry behavior:** Calls `extractRequests()` again for the same document. Shows extraction loading state.

**Upload new behavior:** Opens the native file picker (accept `.pdf`). On selection, uploads to Vercel Blob and triggers the full upload-classify-extract pipeline for the new file.

---

### Auto-Naming Feedback (D-09, D-10)

**When:** Extraction completes and returns a case name, and the case currently has no name (name is null).

**Visual:** The case name heading at the top of the case detail page updates from "Untitled Case" (text-secondary, italic) to the extracted name (e.g., "Swan v. Dollar Tree") in text-primary, weight 600. No transition animation. No toast notification. The change is visible because the page revalidates after extraction.

**If case already has a name:** No change. Auto-name is silently skipped (D-10).

---

### Dashboard Updates

No new dashboard UI in Phase 3. The upload flow on the dashboard creates a case and navigates to `/case/[id]`, where extraction happens. The sidebar case entry may update its name if auto-naming fires.

---

### File Row Status Extension

**New status for `FileUploadState`:** `"extracting"`

**Visual in file row (only visible briefly during upload-classify-extract pipeline):**
- Status area (right-aligned): "Extracting..." text (14px, text-secondary) with lucide `Loader2` spinning icon (14px, accent color)
- Same position and styling as the existing "Classifying..." status

This status is transient -- the file row is removed from pending uploads once extraction completes (or fails), and the extracted requests appear in the section below.

## Interaction Contracts

| Interaction | Behavior |
|------------|----------|
| Automatic extraction trigger | After upload and classification, if document type is `discovery_request`, extraction starts automatically. No manual trigger for initial extraction (D-02). |
| Request row click | Toggles between collapsed (100-char preview) and expanded (full text) for rows with text exceeding 100 characters. Rows with short text are not expandable. |
| Re-extract click | Disables Re-extract button. Shows spinning RefreshCw icon. Clears existing request rows and shows extraction loading state. On completion, replaces with new results. On failure, shows error state. |
| Sub-type badge click | Opens select dropdown with "RFP" and "Interrogatory" options. Selection fires server action to update document sub-type. Badge updates optimistically. |
| Generate Responses click | Disabled in Phase 3. No action. Visual feedback: cursor not-allowed, 50% opacity. |
| Retry click (error state) | Same as re-extract: calls extraction server action for the same document. |
| Upload new click (error state) | Opens native file picker. Uploads new PDF to Vercel Blob and replaces the failed document. Full upload-classify-extract pipeline runs for new file. |
| Auto-naming | After extraction, if case name is null, the heading updates to the extracted case name on page revalidation. No toast. No animation. |
| Page revalidation | `revalidatePath(/case/[id])` called after extraction completes. Server component re-renders with new extracted requests data. |
| Responsive | Not required for v1. Minimum supported width: 1024px. |

## Accessibility

| Concern | Implementation |
|---------|---------------|
| Focus indicators | 2px ring in `--ring` (#C8653A40) with 2px offset on all interactive elements (carries forward) |
| Request row keyboard | Each request row is focusable (`tabIndex={0}`). Enter/Space toggles expand/collapse. `role="button"` on expandable rows. `aria-expanded="true/false"` on expandable rows. |
| Request list structure | Container has `role="list"`. Each request row has `role="listitem"`. |
| Extraction loading | Skeleton area has `aria-busy="true"` and `aria-label="Extracting requests from document"`. Status text "Extracting requests..." is in an `aria-live="polite"` region. |
| Extraction error | Error container has `role="alert"`. Error message is immediately announced. Retry button receives focus after error appears. |
| Sub-type override | Select has accessible label "Discovery type for [document filename]". Keyboard navigable via Base UI Select. |
| Re-extract button | `aria-label="Re-extract requests from document"`. Disabled state communicated via `aria-disabled`. |
| Generate Responses button | `aria-disabled="true"` with `aria-label="Generate Responses - coming soon"`. |
| Color contrast | All text/background combinations maintain WCAG AA (4.5:1). Text-secondary (#78736A) on white (#FFFFFF) = 4.5:1. |
| Screen reader: extraction complete | `aria-live="polite"` region announces "{N} requests extracted" when extraction finishes. |
