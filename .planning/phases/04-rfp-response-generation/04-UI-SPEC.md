# Phase 4 — UI Design Contract

> Visual and interaction contract for RFP Response Generation phase: enabling the "Generate Responses" button, streaming progress indicator, inline generated response display beneath each extracted request, re-generate flow, missing complaint limitation banner, and partial failure recovery UI. ### Pattern-Specific Visual Indicators

Each generated response displays a pattern badge using existing `--muted` and `--muted-foreground`:

| Pattern | Badge Text | Badge Style |
|---------|-----------|-------------|
| produced_all | "Produced All" | `badge` secondary variant, default styling |
| no_such_documents | "No Such Documents" | `badge` secondary variant, default styling |
| objection | "Objection" | `badge` secondary variant, default styling |
| cross_reference | "Cross-Reference" | `badge` secondary variant, default styling |

All badges use the same secondary variant to keep the UI calm. The response text itself communicates the meaning -- color-coding patterns is unnecessary and would violate the 10% accent budget.

## Page Contracts

### Case Detail Page (`/case/[id]`) -- Extended

**Layout:** Unchanged. Same protected layout with sidebar. Main content max-width 640px centered. 48px top padding.

**Existing content (unchanged):** Back link, case name heading, case metadata, documents section with document list, extracted requests section(s), danger zone.

**Phase 4 modifications to Extracted Requests section:**

The `ExtractedRequests` component gains three new capabilities:
1. The "Generate Responses" button becomes functional
2. Generated responses appear inline below each request row
3. A "Re-generate" button replaces "Generate Responses" after generation completes

#### Limitation Banner (No Complaint Uploaded)

**When:** The case has no document with `type === "complaint"` and generation is about to start or has completed.

**Position:** Immediately above the "Generate Responses" button. 16px bottom margin to the button.

**Visual:**
- Container: rounded border (1px `--border`), background `--muted` (#F0EDE6), padding 16px
- Layout: horizontal flex, items-start, 8px gap
- Left: lucide `Info` icon (16px, `--muted-foreground`)
- Right: text block
  - Line 1 (14px, weight 400, `--muted-foreground`): "Responses generated without complaint context."
  - Line 2 (14px, weight 400, `--muted-foreground`): "Upload a complaint and re-generate for better results."

**Behavior:** Banner persists after generation completes. Disappears when a complaint document is added to the case (after page revalidation).

#### Generate Responses Button (Enabled State)

**Position:** Below the last request row (or below the limitation banner if present). Full width. 16px top margin.

**Visual (now enabled):**
- Primary button variant, accent background (#C8653A), white text "Generate Responses" (16px, weight 600), 44px height
- Full width of request list container
- Hover: `bg-primary/80` (standard primary button hover from shadcn)
- No `cursor-not-allowed`, no `opacity-50`, no `disabled` attribute

**Click behavior:** Triggers generation for all extracted requests in the case. Button transitions to generating state.

#### Generating State (Streaming Progress)

**When:** After clicking "Generate Responses". Persists until all responses complete or an error occurs.

**Visual changes to the section:**

1. **Button replacement:** The "Generate Responses" button is replaced by a progress area:
   - Progress label left: "Generating responses..." (14px, weight 400, `--muted-foreground`)
   - Progress value right: "{N} of {total}" (14px, weight 400, `--muted-foreground`, tabular-nums)
   - Below label row: `Progress` component (shadcn), 4px height, full width
     - Track: `--muted` background
     - Indicator: `--primary` (#C8653A) fill, width = (completedCount / totalRequests) * 100%
   - Below progress bar, 8px margin: lucide `Loader2` (14px, spinning, `--primary`) + "This may take up to a minute..." (14px, weight 400, `--muted-foreground`), centered

2. **Request rows with skeleton responses:** As generation streams, each request row that has not yet received a response shows a skeleton placeholder beneath it:
   - Skeleton block: no top border-radius on top edge (connects visually to request row above), border-left and border-right and border-bottom (1px `--border`), padding 16px, background `--card`
   - Inside: skeleton shimmer bar, 100% width, 14px height, rounded. Second bar below at 80% width.

3. **Request rows with completed responses:** As each response arrives from the SSE stream, its skeleton is replaced by the actual generated response block (described below).

#### Generated Response Block (Inline Below Each Request)

**Position:** Directly below its corresponding request row. No gap -- the response block's top edge connects to the request row's bottom edge, creating a visual parent-child relationship.

**Visual:**
- Container: no top border-radius (connects to request row), border-left and border-right and border-bottom only (1px `--border`), padding 16px, background `--card`
- Top row (horizontal flex, items-center):
  - Pattern badge (badge component, secondary variant, 14px text): "Produced All", "No Such Documents", "Objection", or "Cross-Reference"
  - If pattern is `objection` and `objection_types` is not null: additional badges after 4px gap, one per objection type, same secondary variant, text: "Privilege", "Overbroad", "Premature", or "Compound"
  - If pattern is `cross_reference`: additional text after 4px gap (14px, weight 400, `--muted-foreground`): "See Request No. {N}"
- Response text (16px top margin from badge row):
  - Text: 16px, weight 400, `--foreground`, line-height 1.5
  - Preserves paragraph breaks from the AI output. Each paragraph separated by 16px margin.
  - For objection responses: the "Without waiving any objection" bridge phrase and subsequent substantive response render as natural paragraph flow -- no special visual treatment.

**Interaction:** Response blocks are read-only in Phase 4. Editing comes in Phase 6 (Review UI).

#### Post-Generation State (All Responses Complete)

**When:** All responses have been received and persisted. Generation stream has ended.

**Visual changes:**

1. **Progress area removed.** Replaced by "Re-generate" button.

2. **Toast notification:** `sonner` success toast: "All {N} responses generated" (auto-dismiss after 4 seconds).

3. **Re-generate button:**
   - Position: where "Generate Responses" button was. Full width. 16px top margin from last response block.
   - Visual: outline button variant (not primary -- lower visual prominence since generation is already done). Text: "Re-generate All Responses" (14px, weight 600). lucide `RefreshCw` icon (14px) left of text.
   - 44px height.

4. **All request rows now show their response blocks** beneath them. The collapsed/expanded state of the request text is independent of the response block visibility -- the response block is always visible when a response exists.

#### Re-generate Confirmation Dialog

**When:** User clicks "Re-generate All Responses".

**Visual:** `AlertDialog` component (already installed, same pattern as "Delete Case" dialog):
- Title: "Re-generate all responses?"
- Description: "This will delete all existing responses and generate new ones. Any previous responses will be permanently replaced."
- Cancel button: "Cancel" (outline variant)
- Confirm button: "Re-generate" (primary variant, accent background)

**Behavior on confirm:** Deletes all existing responses (D-09), transitions to generating state. Same streaming progress flow as initial generation.

#### Generation Error State

**When:** The SSE stream errors out (Claude API failure, network timeout, Vercel function timeout).

**Partial completion case (some responses received before failure):**

The responses that arrived successfully remain displayed. The progress area is replaced by an error + continue area:

- Error container: rounded border (1px `--border`), padding 16px, background `--card`
  - Top row: lucide `AlertCircle` (16px, `--destructive`) + error message (14px, weight 400, `--foreground`): "Generation stopped after {N} of {total} responses. Some responses were saved."
  - Below, 16px margin: "Continue Generation" button (outline variant, 14px weight 600, 44px height, full width). lucide `Play` icon (14px) left of text.

**Zero completion case (error before any responses):**

- Error container: same as extraction error from Phase 3 but with different copy
  - lucide `AlertCircle` (24px, `--destructive`), centered
  - Message (14px, weight 400, `--foreground`, centered): "Could not generate responses. The AI service may be temporarily unavailable."
  - 16px below: two buttons centered, 8px gap:
    - "Retry" button (outline variant, 14px): lucide `RefreshCw` icon (14px) + "Retry" text
    - The retry button triggers the same generation flow from the beginning

**Continue Generation behavior:** Sends only the un-generated requests to the Route Handler. New responses stream in and fill the remaining skeleton/empty slots. Progress counter resumes from where it left off (e.g., "13 of 33 generated" if 12 were saved).

## Interaction Contracts

| Interaction | Behavior |
|------------|----------|
| Generate Responses click | Starts SSE stream to `/api/generate`. Button replaced by progress area. Responses stream in one by one. Each response persisted to DB as it arrives (or bulk at end per hybrid approach). |
| Progress updates during streaming | Progress counter increments as responses complete. Progress bar fills proportionally. Skeleton placeholders beneath request rows replace with actual response blocks as each completes. |
| Generation complete | Progress area replaced by "Re-generate All Responses" button. Success toast shown. All responses visible inline beneath their request rows. |
| Re-generate click | AlertDialog confirmation. On confirm: deletes all existing responses, restarts generation from scratch. Same streaming flow. |
| Continue Generation click (after partial failure) | Resumes generation for un-generated requests only. Progress counter starts from saved count. No confirmation dialog (not destructive -- adds to existing, does not replace). |
| Retry click (after zero failure) | Restarts generation from scratch. Same flow as initial "Generate Responses" click. |
| Request row expand/collapse | Unchanged from Phase 3. Response block remains visible regardless of request text expand state. |
| Page refresh with existing responses | Generated responses load from database and display inline beneath their request rows. No re-generation triggered. "Re-generate All Responses" button shown. |
| Limitation banner visibility | Shown when no complaint document exists on the case. Persists through generation and after completion. Removed on page revalidation after complaint upload. |
| Responsive | Not required for v1. Minimum supported width: 1024px. |

## Accessibility

| Concern | Implementation |
|---------|---------------|
| Focus indicators | 2px ring in `--ring` (#C8653A40) with 2px offset on all interactive elements (carries forward) |
| Generate Responses button | Standard button semantics. `aria-label="Generate responses for all extracted requests"` when enabled. Focus ring on tab. |
| Progress indicator | `Progress` component from shadcn uses `role="progressbar"` with `aria-valuenow`, `aria-valuemin`, `aria-valuemax`. Label "Generating responses..." associated via `ProgressLabel`. |
| Generation complete announcement | `aria-live="polite"` region announces "All {N} responses generated" when streaming finishes. |
| Pattern badges | Decorative (info is in the response text). `aria-hidden="true"` on badge container. |
| Response text blocks | Standard text content. No special ARIA needed. Associated with request via DOM proximity. |
| Error states | Error container has `role="alert"`. Error message immediately announced. "Continue Generation" or "Retry" button receives focus after error appears. |
| Re-generate dialog | `AlertDialog` provides `role="alertdialog"` with focus trapping. Cancel button receives initial focus. Escape key dismisses. |
| Limitation banner | `role="status"`. `aria-live="polite"` for screen reader announcement. lucide `Info` icon has `aria-hidden="true"`. |
| Color contrast | All text/background combinations maintain WCAG AA (4.5:1). Progress bar color (#C8653A on #F0EDE6) meets 3:1 minimum for non-text elements. |
| Keyboard navigation | Tab order: request rows, then response blocks (read-only), then Generate/Re-generate button. During generation, progress area is not focusable (informational only). |
